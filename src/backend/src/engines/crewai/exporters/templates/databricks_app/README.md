# {{DISPLAY_NAME}}

A **CrewAI** crew packaged as a conversational Databricks App. The app ships with a built-in chat UI and
also exposes an MLflow `AgentServer` endpoint (`/invocations`) so you can query the crew from anywhere
(your website, a mobile app, the Databricks OpenAI client, or curl).

The crew is defined in `config/agents.yaml` + `config/tasks.yaml` and run behind MLflow's
**ResponsesAgent** interface (`agent_server/agent.py`). Each chat turn maps the user's latest message to
the crew's input (`INPUT_KEY`, e.g. `topic`), calls `crew.kickoff(inputs=...)`, and streams back the
result. The Responses I/O format is documented here:
https://mlflow.org/docs/latest/genai/flavors/responses-agent-intro/

> Generated by **Kasal**. To change the crew, edit `config/agents.yaml`/`config/tasks.yaml`, or the
> `# GENERATED` block + `MODEL_OVERRIDE`/`INPUT_KEY` in `agent_server/agent.py`.

## Quick start

Run `uv run quickstart` to set up your local environment (verifies `uv`/Node/Databricks CLI, configures
auth, creates + links an MLflow experiment) and start the agent server + chat app:

```bash
uv run quickstart
```

After setup, start the agent server and chat app locally with:

```bash
uv run start-app
```

This serves the chat UI and the agent at http://localhost:8000.

## Manual local development loop

1. **Install** `uv`, `nvm` (Node 20 LTS), and the Databricks CLI.
2. **Authenticate** to Databricks:
   ```bash
   databricks auth login
   ```
   Then set `DATABRICKS_CONFIG_PROFILE` in `.env` (copy `.env.example` to `.env`).
3. **Create + link an MLflow experiment** (done automatically by `uv run quickstart`):
   ```bash
   DATABRICKS_USERNAME=$(databricks current-user me | jq -r .userName)
   databricks experiments create-experiment /Users/$DATABRICKS_USERNAME/agents-on-apps
   ```
   Put the experiment ID in `MLFLOW_EXPERIMENT_ID` in `.env`.
4. **Run** `uv run start-app` and query the crew:

   ```bash
   curl -X POST http://localhost:8000/invocations \
     -H "Content-Type: application/json" \
     -d '{ "input": [{ "role": "user", "content": "AI trends in 2025" }] }'
   ```

   Advanced server options: `uv run start-server --reload`, `--port 8001`, `--workers 4`.

## Modifying your crew

- **Agents/tasks:** edit `config/agents.yaml` and `config/tasks.yaml`. Each agent's `llm`, `role`, `goal`,
  `backstory`, and `tools` come straight from the crew you exported.
- **Model:** set `MODEL_OVERRIDE` in `agent_server/agent.py` (or leave `None` to keep each agent's `llm`).
- **Input mapping:** `INPUT_KEY` controls which crew input the chat message fills (default from your crew).
- **Tools:** add CrewAI/`crewai_tools` tools to `TOOL_MAP` in `agent_server/agent.py`; for
  Databricks-managed tools via MCP, add `(name, url)` pairs to `MCP_SERVERS` (see the `add-tools` skill).
- **Add dependencies:** `uv add <package>`.

### On-behalf-of (OBO) user authentication

When OBO is enabled (`ENABLE_OBO = True`), Databricks-managed tools/MCP run as the requesting user via the
forwarded `x-forwarded-access-token` (see `agent_server/utils.get_user_workspace_client`). Otherwise they
run as the app's service principal — grant it access to any resources in `databricks.yml`.

## Evaluating your crew

```bash
uv run agent-evaluate
```

Edit `agent_server/evaluate_agent.py` to set your dataset and scorers, then open the MLflow experiment to
inspect results.

## Deploying to Databricks Apps

This template supports two paths:

- **One-click from Kasal** — the *Deploy to Databricks Apps* button in Kasal's Export dialog creates the
  app, uploads this project, deploys, and starts it under your identity. No local CLI needed.
- **Databricks Asset Bundles (DABs)** — `databricks.yml` defines the app + experiment resources.

> `app.yaml` is read by `databricks apps deploy` (and the Kasal one-click path); `databricks.yml`'s
> `config:` section takes precedence for DABs. Keep env vars / the start command in sync across both.

Manual DABs deploy:

```bash
uv run preflight                 # smoke-test locally first
databricks bundle validate
databricks bundle deploy
databricks bundle run {{BUNDLE_NAME}}   # bundle run is required to (re)start the app with new code
```

### Querying the deployed agent

Use a Databricks OAuth token (PATs are not supported for querying agents on Apps):

```python
from databricks.sdk import WorkspaceClient
from databricks_openai import DatabricksOpenAI

w = WorkspaceClient()
client = DatabricksOpenAI(workspace_client=w)
response = client.responses.create(
    model="apps/{{APP_NAME}}",
    input=[{"role": "user", "content": "AI trends in 2025"}],
)
print(response)
```

### Common issues

- **`bundle deploy` says the app already exists** — bind it:
  `databricks bundle deployment bind {{BUNDLE_NAME}} {{APP_NAME}} --auto-approve`, then redeploy. Or delete
  it: `databricks apps delete {{APP_NAME}}`.
- **App runs old code after `bundle deploy`** — you must also run `databricks bundle run {{BUNDLE_NAME}}`.
- **302 when querying** — use an OAuth token, not a PAT.
